# 08-MAX全网素材情报工作台 V0.1 设计文档

版本：V0.1  
定位：本地 Web 工作台 / 抖音单平台先测 / 低风险素材情报链路  
上游：`01-全网情报检索规则-V0.1`、`02-关键词与来源池-V0.1`  
下游：`03-MAX外部情报日报模板-V0.1`、`04-MAX外部情报日报`、`05-选题规则-V2.3`、`06-MAX选题库调试记录`、`07-MAX选题库`

---

# 0. 先反驳一个危险假设

不要把本工作台想成：

```text
搭一个爬虫
↓
全平台自动抓
↓
AI 自动筛
↓
自动写日报
↓
自动生成选题
```

这个想法在 V0.1 阶段不成立。

原因不是工程师写不出来，而是它混淆了四件事：

```text
平台是否允许
账号是否安全
数据是否稳定
素材是否值得进入 MAX 系统
```

尤其是抖音、小红书、微博等平台，真正的难点不只是页面解析，而是协议、登录态、验证码、访问频率、字段暴露、风控和账号风险。

所以 V0.1 的正确目标不是“自动化率最高”，而是：

```text
先跑通一个低风险、可人工审核、可回滚、不会污染正式知识库的素材情报闭环。
```

# 1. 工作台定位

## 1.1 它是什么

`MAX全网素材情报工作台` 是 CHEN 操盘侧的本地 Web 工作台。

它负责把外部平台上的素材，以统一字段进入本地素材池，并经过 AI 初筛，形成：

```text
候选素材池
日报草稿
选题卡片草稿
```

它服务的是：

```text
外部现实信号
↓
MAX 可判断素材
↓
CHEN 人工审核
↓
正式日报 / 正式选题库
```

## 1.2 它不是什么

它不是全网爬虫平台。

它不是自动搬运系统。

它不是 MAX 知识库。

它不是 RAG。

它不是选题库本体。

它不是内容生产系统。

它不能替代 CHEN / MAX 判断。

一句话：

```text
工作台只负责把外部素材变成可审阅的情报候选，不负责把候选自动变成 MAX 正式资产。
```

# 2. V0.1 做什么 / 不做什么

## 2.1 V0.1 做什么

V0.1 只做一个最小闭环：

```text
抖音素材输入
↓
本地统一字段归档
↓
AI 按 01/02/03/05 初筛
↓
生成候选素材
↓
生成日报草稿
↓
生成选题卡片草稿
↓
人工确认是否进入 04 / 06 / 07
```

V0.1 优先支持四类输入：

```text
1. 已知抖音视频链接
2. 用户手动粘贴的视频文案 / 评论区摘录 / 热度数据
3. 用户上传的视频截图 / 评论截图，走 OCR
4. 用户上传音频 / 视频转写稿，走 ASR 或人工粘贴转写
```

V0.1 可以做：

```text
素材录入
字段标准化
重复素材识别
来源池管理
关键词池管理
人工采集任务列表
低频提醒式自动发现任务
OCR / 转写 / 文案提取
AI KEEP / BUFFER / DROP 初筛
日报草稿生成
选题卡片草稿生成
人工审核状态流转
导出 Markdown
```

## 2.2 V0.1 不做什么

V0.1 不做：

```text
不做全平台并发爬取
不做绕验证码
不做反风控
不做模拟真人滑动
不做批量登录态维持
不做主账号自动化访问
不做评论区大规模抓取
不做视频批量下载
不做高频搜索页抓取
不做自动写入正式 04 日报
不做自动写入正式 07 选题库
不把外部素材写入 MAX Wiki / RAG
```

## 2.3 V0.1 的成功标准

V0.1 成功，不看抓了多少。

只看四件事：

```text
1. 手动或半自动输入 10-30 条抖音素材后，字段是否稳定。
2. AI 初筛能否按 01 规则稳定区分 KEEP / BUFFER / DROP。
3. 日报草稿能否贴近 03 模板，减少 CHEN 整理成本。
4. 选题卡片草稿能否进入 06 调试，而不是直接污染 07。
```

如果 V0.1 连 30 条手动素材都处理不好，继续追求自动抓取只会放大垃圾。

# 4. 自动采集和手动采集流程

## 4.1 必须拆开的两件事

自动采集要拆成两类：

```text
自动发现：从关键词 / 账号 / 热榜 / 来源池里找到候选素材。
自动提取：对已知链接 / 视频 / 截图 / 文案 / 评论摘录做结构化提取。
```

V0.1 的重心不是自动发现，而是自动提取。

原因很简单：

```text
自动发现更容易碰平台风控。
自动提取更接近内部工作流，风险低、价值高、可控性强。
```

## 4.2 手动采集流程

```text
CHEN / MAX / AI 发现素材
↓
复制链接 / 截图 / 文案 / 评论摘录 / 热度数据
↓
进入工作台「新增素材」
↓
选择平台：抖音
↓
选择输入类型：链接 / 文案 / 截图 / 转写 / 评论摘录
↓
保存到原始采集区
↓
自动字段标准化
↓
进入候选素材池
↓
AI 初筛 KEEP / BUFFER / DROP
↓
生成日报草稿或选题卡片草稿
↓
人工审核
```

手动采集不是低级方案。

在 V0.1，手动采集反而是最优方案，因为它把平台风险压低，同时可以快速验证字段、规则和 AI 初筛质量。

## 4.3 自动采集流程

V0.1 的自动采集只做低风险版本：

```text
来源池 / 关键词池
↓
生成每日人工采集任务
↓
提醒 CHEN 去指定平台手动查看
↓
CHEN 把值得看的链接 / 截图 / 文案丢回工作台
↓
工作台自动提取和初筛
```

也就是说，V0.1 的“自动发现”更准确地说是：

```text
自动生成采集清单，而不是无人值守爬平台。
```

只有当某个平台存在稳定、合法、低风险、无需绕过限制的公开接口时，才允许进入下一阶段的自动发现。

# 5. 抖音 V0.1 先测哪些能力

## 5.1 抖音 V0.1 的原则

抖音 V0.1 不做“抖音爬虫”。

抖音 V0.1 做：

```text
抖音素材进入 MAX 工作台的本地处理链路。
```

公开平台资料显示，抖音官方视频数据能力更偏向授权后的账号视频数据、统计数据和开放平台能力；它不是给任意第三方无限搜索、批量抓取全站内容的通道。因此 V0.1 不应假设存在一个稳定官方通道可以任意抓取全抖音素材。

## 5.2 第一批测试能力

### 能力 1：已知链接录入

输入：

```text
抖音视频链接
视频标题
来源账号
发布时间
抓取时间
点赞 / 评论 / 收藏 / 分享
视频文案
评论摘录
```

输出：

```text
标准化素材记录
```

说明：

```text
链接只作为来源凭证和去重依据。
不要在 V0.1 里幻想仅凭链接就稳定自动拿到全部字段。
字段拿不到，就允许人工补。
```

### 能力 2：截图 OCR

输入：

```text
视频页面截图
评论区截图
热榜截图
账号主页截图
```

输出：

```text
OCR 文本
疑似字段
待人工确认字段
```

说明：

```text
OCR 结果必须标注置信度。
不能把 OCR 识别结果直接当事实写入正式日报。
```

### 能力 3：转写稿处理

输入：

```text
用户粘贴转写稿
视频字幕
音频转写结果
```

输出：

```text
内容摘要
关键观点
可观察场景
MAX 相关判断
```

说明：

```text
V0.1 不优先做视频下载。
如果用户已经有音频、视频或转写文本，工作台再处理。
```

### 能力 4：评论摘录结构化

输入：

```text
用户手动粘贴高赞评论
评论区截图 OCR
重复出现的评论语言
```

输出：

```text
评论区核心声音
争议点
共识点
用户原话摘录
```

说明：

```text
评论区在 V0.1 只做摘录，不做大规模抓取。
```

### 能力 5：AI 初筛

输入：

```text
标准化素材记录
```

处理规则：

```text
01 判断 KEEP / BUFFER / DROP
02 记录命中的关键词 / 来源入口
03 生成日报草稿字段
05 生成选题卡片草稿字段
```

输出：

```text
候选素材状态
日报草稿
选题卡片草稿
需要补充的信息
```

# 6. 各平台适配器统一字段

平台适配器必须输出同一种 `Material` 字段结构。

## 6.1 基础来源字段

```text
material_id：本地唯一 ID
platform：平台，douyin / xiaohongshu / weibo / bilibili / youtube / x / instagram
source_type：search / account / hotlist / manual / link / screenshot / transcript
source_url：原始链接
canonical_url：清洗后的标准链接
source_account_name：来源账号名称
source_account_id：来源账号 ID，如可得
source_account_url：来源账号主页，如可得
publish_time：内容发布时间，如可得
captured_at：本地采集时间
captured_by：manual / system / import
```

## 6.2 内容字段

```text
title：原始标题或内部识别标题
raw_text：原始文案 / OCR / 转写文本
summary：素材摘要
media_type：video / image / text / audio / mixed
duration：视频或音频时长，如可得
screenshots：截图文件路径
transcript：转写稿
ocr_text：OCR 结果
comment_samples：评论摘录
```

## 6.3 热度字段

```text
like_count：点赞数
comment_count：评论数
collect_count：收藏数
share_count：分享数
repost_count：转发数
view_count：播放量，如可得
hot_rank：热榜位置，如可得
engagement_note：人工补充热度说明
metric_captured_at：热度数据采集时间
metric_confidence：exact / approximate / screenshot / manual / unknown
```

## 6.4 检索字段

```text
matched_keyword：命中的关键词
matched_source_pool：命中的来源池
search_query：当次检索词
discovery_path：如何发现这条素材
related_topic：价值感 / 展示面 / 情感反馈 / 审美误判 / 阶层规则等
```

## 6.5 AI 初筛字段

```text
ai_status：KEEP / BUFFER / DROP
ai_reason：为什么这么判断
max_relevance：和 MAX 主线的关系
user_relevance：目标用户为什么会关心
possible_angles：初步 MAX 可讲视角
missing_info：缺失信息
risk_flags：风险标记
review_required：是否必须人工复核
```

## 6.6 人工审核字段

```text
review_status：unreviewed / approved_for_daily_draft / approved_for_topic_draft / rejected / needs_more_info
reviewer：审核人
reviewed_at：审核时间
review_note：审核意见
final_destination：none / daily_draft / topic_draft / official_daily / debug_log / official_topic_library
```

# 7. 五个素材区的边界

## 7.1 原始采集区

可以自动写。

存放：

```text
链接
截图
文案
转写稿
评论摘录
热度截图
平台原始字段
```

要求：

```text
只保存事实输入。
不做价值判断。
不覆盖原始内容。
允许脏、乱、缺字段。
```

## 7.2 候选素材池

可以自动写。

存放：

```text
已经结构化的素材
AI 初筛结果
KEEP / BUFFER / DROP
缺失信息
风险标记
```

要求：

```text
候选池不是日报。
候选池不是选题库。
DROP 也可以保留，用于反向训练检索规则。
```

## 7.3 日报草稿区

可以自动生成，但必须人工确认。

存放：

```text
按 03 模板生成的日报条目草稿
```

要求：

```text
只能生成草稿。
不能自动追加到 04-MAX外部情报日报。
```

## 7.4 选题卡片草稿区

可以自动生成，但必须人工确认。

存放：

```text
按 05 规则压缩出的候选 MAX 可口喷卡片
```

要求：

```text
卡片草稿优先进入 06 调试。
只有经过调试、确认可口喷，才允许进入 07。
```

## 7.5 正式写入区

不能自动写入。

正式写入区包括：

```text
04-MAX外部情报日报
06-MAX选题库调试记录
07-MAX选题库
```

写入规则：

```text
04：只写人工确认后的 KEEP / BUFFER 日报条目。
06：只写发生过真实调试、纠偏、规则沉淀的过程。
07：只写 MAX 可以直接读、直接判断、直接口喷的材料卡片。
```

硬规则：

```text
工作台可以导出 Markdown。
工作台不能自行覆盖正式文件。
工作台不能绕过人工确认。
工作台不能把候选素材自动写入 Wiki / RAG。
```

# 8. 账号、频率、登录态、风控、安全边界

## 8.1 账号边界

```text
不使用 MAX 主账号做自动化采集。
不把主账号 Cookie 写入代码或配置。
不在本地明文保存任何平台登录态。
不做多账号轮换。
不购买或使用异常账号池。
```

如果未来需要测试登录态能力，必须单独使用：

```text
测试账号
低频
人工操作
明确记录
可随时停用
```

## 8.2 频率边界

V0.1 不做高频抓取。

建议频率：

```text
手动采集：不限，但由人决定。
自动生成采集任务：每天 1-2 次。
自动处理已录入素材：本地即时处理。
对平台发起请求：默认不做；如做，必须低频、可配置、可关闭。
```

## 8.3 登录态边界

```text
V0.1 默认不接入平台登录态。
不抓取需要登录后才能查看的私人内容。
不绕过平台限制访问隐藏字段。
```

## 8.4 风控边界

禁止做：

```text
验证码识别
验证码绕过
设备指纹伪造
代理池
模拟真人行为
批量请求
反检测脚本
无限重试
```

允许做：

```text
人工复制链接
人工截图
人工粘贴评论
人工上传转写
本地 OCR
本地 AI 分析
人工审核
```

## 8.5 数据安全边界

```text
本地数据默认不公开。
原始素材与正式资产分开存。
AI 输出必须标记为草稿。
截图、转写、评论摘录保留来源。
所有正式写入都保留审核记录。
```

# 9. 如何连接 01/02/03/04/05/06/07 链路

## 9.1 连接 01：全网情报检索规则

工作台的 `analyzer` 必须调用 01 的判断标准：

```text
价值感破绽
极小动作暴露身份
情感关系中的价值表达
具体情感关系事件
阶层与名利场规则
审美与生活方式误判
公共热点里的 MAX 视角
```

输出：

```text
KEEP / BUFFER / DROP
ai_reason
missing_info
risk_flags
```

## 9.2 连接 02：关键词与来源池

工作台维护：

```text
关键词列表
来源账号列表
热榜入口
有效 / 无效检索词记录
```

V0.1 中，02 不直接驱动自动爬虫，而是驱动：

```text
每日人工采集任务
素材发现路径记录
关键词命中记录
来源池有效性复盘
```

## 9.3 连接 03：日报模板

日报草稿必须按 03 输出：

```text
情报标题
事件 / 现象 / 故事梗概
热度与来源
评论区核心声音
初步 MAX 可讲视角
可选补充信息
KEEP / BUFFER 标记
待补信息
```

## 9.4 连接 04：外部情报日报

工作台只能生成：

```text
04 待写入草稿
```

不能自动追加。

人工确认后，才写入：

```text
04-MAX外部情报日报
```

## 9.5 连接 05：选题规则

选题卡片草稿必须按 05 压缩：

```text
材料压缩
用户相关性判断
MAX 可讲性判断
切口能量校准
推荐切入点
```

## 9.6 连接 06：选题库调试记录

所有不稳定、不确定、不够准的卡片，优先进入 06，而不是 07。

进入 06 的条件：

```text
切口能量不对
太像说教
太像情感号
太负向
太抽象
没有提供价值
MAX / CHEN 有纠偏
需要沉淀新规则
```

## 9.7 连接 07：MAX选题库

07 只接收最终通过的卡片。

进入 07 的最低标准：

```text
MAX 可以直接读
MAX 可以直接判断
MAX 可以直接口喷
```

如果只是“AI 觉得能讲”，不能进入 07。

# 10. 本地 Web 工作台页面设计

V0.1 不追求复杂 UI，只需要完成五个页面。

## 10.1 首页：今日工作台

展示：

```text
今日新增素材数
待审核素材数
KEEP / BUFFER / DROP 数量
待补信息
日报草稿数
选题卡片草稿数
```

## 10.2 新增素材

支持：

```text
粘贴链接
粘贴文案
上传截图
粘贴评论摘录
粘贴转写稿
填写热度数据
选择关键词 / 来源池
```

## 10.3 候选素材池

支持：

```text
按平台筛选
按 KEEP / BUFFER / DROP 筛选
按关键词筛选
按来源账号筛选
按待补信息筛选
查看 AI 判断理由
人工修改状态
```

## 10.4 草稿区

分为：

```text
日报草稿
选题卡片草稿
```

支持：

```text
预览 Markdown
编辑草稿
标记进入 04 / 06 / 07
导出 Markdown
```

## 10.5 设置区

管理：

```text
关键词池
来源池
平台适配器开关
AI Prompt 模板
字段映射
安全频率配置
```

# 11. 数据存储建议

V0.1 推荐用本地轻量存储。

优先级：

```text
SQLite：存结构化素材、状态、审核记录
本地文件夹：存截图、转写稿、导出 Markdown
JSON/YAML：存关键词池、来源池、Prompt 配置
```

不建议 V0.1 一开始上复杂数据库、云服务、队列系统。

原因：

```text
V0.1 的核心风险不是性能，而是边界和质量。
```

# 12. AI 初筛 Prompt 的输入输出约束

## 12.1 输入

```text
标准化素材字段
01 检索规则摘要
02 命中关键词 / 来源
03 日报模板字段
05 选题规则摘要
```

## 12.2 输出

必须输出：

```text
ai_status：KEEP / BUFFER / DROP
ai_reason：判断理由
daily_draft：日报草稿
topic_card_draft：选题卡片草稿
missing_info：缺失信息
risk_flags：风险
review_questions：给 CHEN 的审核问题
```

## 12.3 输出纪律

AI 不能输出：

```text
最终口播稿
最终标题
自动入库结论
“已进入 07”这类越权状态
```

AI 必须承认：

```text
这是草稿。
需要人工确认。
缺字段时不能装作完整。
```

# 13. 后续交给 Codex 的最小施工路线

## 阶段 1：建立本地工程骨架

目标：

```text
能打开一个本地 Web 工作台。
```

任务：

```text
创建项目目录
搭建前后端
接 SQLite
建立素材表
建立本地文件夹结构
实现首页和新增素材页
```

验收：

```text
可以手动新增一条抖音素材。
刷新后素材仍在。
```

## 阶段 2：实现统一字段和候选素材池

目标：

```text
所有输入都进入统一 Material 字段。
```

任务：

```text
实现 Material schema
实现平台字段 platform=douyin
实现候选素材池列表
实现 KEEP / BUFFER / DROP 手动状态
实现重复链接识别
```

验收：

```text
同一个链接重复录入时能提示。
素材可以按状态筛选。
```

## 阶段 3：接 OCR / 转写 / 文案提取

目标：

```text
截图、文案、转写稿都能进入 raw_text。
```

任务：

```text
上传截图
本地 OCR
粘贴转写稿
评论摘录字段
标记 OCR 置信度
```

验收：

```text
上传评论截图后，能生成可编辑 OCR 文本。
```

## 阶段 4：接 AI 初筛

目标：

```text
根据 01/02/03/05 生成候选判断。
```

任务：

```text
写 analyzer prompt
读取素材字段
输出 KEEP / BUFFER / DROP
生成日报草稿
生成选题卡片草稿
记录缺失信息
```

验收：

```text
输入 10 条素材，AI 能生成可审阅草稿，并且不会自动写正式文件。
```

## 阶段 5：实现人工审核与 Markdown 导出

目标：

```text
CHEN 可以把确认后的草稿导出，手动写入 04 / 06 / 07。
```

任务：

```text
审核状态流转
草稿编辑
Markdown 预览
Markdown 导出
审核日志
正式写入前确认提示
```

验收：

```text
一条 KEEP 素材可以导出为 03 模板格式的日报条目。
一条候选卡片可以导出为 07 模板格式。
系统不自动改 04 / 07。
```

## 阶段 6：再评估是否做低频自动发现

目标：

```text
只在前五阶段稳定后，评估自动发现。
```

允许尝试：

```text
每日采集任务生成
来源池提醒
关键词清单推送
公开热榜链接记录
```

不允许直接升级为：

```text
无人值守全站抓取
登录态抓取
高频评论抓取
多账号自动化
```

# 14. V0.1 的第一条 Codex 施工指令

```text
请搭建「MAX全网素材情报工作台 V0.1」本地 Web 工程。

当前只实现抖音单平台的手动素材录入闭环，不做自动爬虫。

必须实现：
1. 本地 Web 页面
2. SQLite 数据库
3. Material 统一字段
4. 新增素材页：支持抖音链接、标题、账号、发布时间、热度数据、视频文案、评论摘录、转写稿、截图路径
5. 候选素材池：支持列表、详情、KEEP / BUFFER / DROP 状态、待补信息
6. 草稿区：日报草稿、选题卡片草稿，只生成草稿，不写正式文件
7. Markdown 导出
8. 审核状态：unreviewed / approved_for_daily_draft / approved_for_topic_draft / rejected / needs_more_info

禁止实现：
1. 自动登录抖音
2. Cookie 存储
3. 验证码绕过
4. 反风控
5. 高频抓取
6. 自动写入 04-MAX外部情报日报
7. 自动写入 07-MAX选题库

验收标准：
1. 本地能打开工作台
2. 能新增 3 条抖音素材
3. 能在候选素材池看到并筛选
4. 能生成一条日报草稿 Markdown
5. 能生成一条选题卡片草稿 Markdown
6. 正式 04 / 07 文件不会被程序自动修改
```

# 15. 阶段性结论

V0.1 最重要的不是“抓”，而是“进系统”。

现在真正该验证的是：

```text
一个抖音素材进入本地工作台后，
能不能被稳定结构化，
能不能按 MAX 规则初筛，
能不能生成可审阅草稿，
能不能经过人工确认后再进入正式链路。
```

如果这个闭环成立，再谈自动发现。

如果这个闭环不成立，全自动只会把无效素材、更脏字段、更大的账号风险，一起倒进 MAX 系统。
