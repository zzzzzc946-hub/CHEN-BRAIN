# 07｜CHEN BRAIN仓库协作规则

## 一、文档定位

本文档用于定义 `CHEN BRAIN` 仓库的基础协作规则。

本仓库首先是 Obsidian Markdown 知识库，其次才通过 GitHub 进行版本管理和跨工具协同。

后续 ChatGPT、Codex、Obsidian、GitHub 参与本项目时，必须优先尊重本文档约定。

------------------------------------------------------------------------

## 二、仓库定位

当前仓库根目录为：

    CHEN BRAIN

当前阶段版本定位为：

    CHEN BRAIN｜V1.0

在 V1.0 阶段，本仓库核心服务于：

    MAX IP项目

因此，本仓库不作为普通代码项目管理，不创建 `src`、`app`、`config`、`package.json` 等程序项目结构，除非 CHEN 后续明确启动对应技术项目。

------------------------------------------------------------------------

## 三、一级目录保护规则

以下五个一级目录名称不得擅自修改：

    00｜MAX IP项目总控台
    01｜Inbox｜md转换
    02｜MAX顶层思维系统
    03｜CHEN操盘手系统
    04｜MAX × CHEN AI协同系统

任何 AI、自动化工具或协作者不得在未经 CHEN 确认的情况下，对上述目录进行删除、移动、重命名或合并。

------------------------------------------------------------------------

## 四、核心文档保护规则

各系统中的 `00｜...说明.md` 和项目总纲类文件属于系统入口文档。

修改这些文档前，应先明确：

    修改目的
    修改范围
    是否影响系统边界
    是否影响后续 AI 读取顺序

如果只是新增协同规则、操作说明、复盘记录或临时草稿，应优先新增独立文档，不直接改写核心入口文档。

------------------------------------------------------------------------

## 五、Inbox 文件保护规则

`01｜Inbox｜md转换` 是输入、转换和归档区。

该目录中的源文件、已转换文件和归档文件不得擅自删除。

如果后续需要整理 Inbox，只能在 CHEN 确认后进行，并应优先采用：

    保留源文件
    标记处理状态
    新增归档说明
    避免直接删除

------------------------------------------------------------------------

## 六、Git 版本管理规则

适合进入版本管理的内容包括：

    Markdown 文档
    Inbox 中需要保留的源文件
    Obsidian 核心配置
    非敏感的 Obsidian 插件配置
    仓库协作规则文档

不适合进入版本管理的内容包括：

    .DS_Store
    Obsidian workspace 状态文件
    缓存文件
    临时文件
    日志文件
    .env 文件
    API Key
    私钥
    本地证书

------------------------------------------------------------------------

## 七、协作原则

本仓库协作遵循以下原则：

    先读总纲，再做判断
    先确认边界，再修改文件
    优先新增说明，不轻易改写核心文档
    不移动、不重命名、不删除核心目录
    不删除 Inbox 源文件
    不把本地密钥和临时状态提交到 GitHub

------------------------------------------------------------------------

## 八、AI 操作底线

任何 AI 在本仓库中执行文件操作时，必须遵守：

    不擅自删除文件
    不擅自重构目录
    不擅自修改项目定位
    不擅自覆盖 CHEN 的判断
    不把临时缓存和密钥纳入版本管理

AI 可以提出建议、生成草稿、整理结构、辅助执行，但最终判断权属于 CHEN。
